# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Pinfold is an offline KML/KMZ catalogue app for iOS/iPadOS/macOS (Designed for iPad), min **iOS 26**, Liquid Glass UI. Two parts:
- `PinfoldCore/` — a standalone Swift Package (no UIKit/SwiftUI/SwiftData): the KML/KMZ parser. Entry point `KMLReader.read(data:) -> ParsedKML`. Depends on ZIPFoundation.
- `App/`, `ShareExtension/`, `AppTests/` — the app, built into an Xcode project **generated by XcodeGen** from `project.yml`.

Design docs and implementation plans live in `docs/superpowers/` (including `docs/superpowers/plans/`, e.g. the review-implementation plan that drove the parser hardening, sync, split-view, search, favorites, tags, and Spotlight work). Test fixtures (sample KML/KMZ) live in `AppTests/Fixtures/` and `PinfoldCore/Tests/PinfoldCoreTests/Fixtures/`.

## Build & test — use the scripts, not raw xcodebuild

- App build: `scripts/build.sh` · App tests: `scripts/test.sh` (filter: `scripts/test.sh -only-testing:PinfoldTests/<Suite>`).
  These target the **iPhone 17 simulator on iOS 26.5 with code signing disabled** — the only known-good destination here (the device/team isn't configured, and only the 26.5 simulator runtime is installed).
- Parser package tests: `cd PinfoldCore && swift test`.

## Formatting & linting

- **Format:** `swiftformat` (config-less, pinned via `.swift-version`). A PostToolUse hook auto-formats each `.swift` file Claude edits — let it own comma/spacing style.
- **Lint:** `swiftlint lint` (config in `.swiftlint.yml`). Keep it at **0 errors**; the existing warnings are a known backlog. `trailing_comma` is disabled there on purpose (SwiftFormat owns comma style). Annotate genuinely-intentional violations inline (e.g. `// swiftlint:disable:next force_try -- <reason>`) rather than loosening a rule globally.

## XcodeGen workflow

- `Pinfold.xcodeproj` is **generated by XcodeGen but committed** (Xcode Cloud builds the checked-in project as-is — there is no `ci_scripts/ci_post_clone.sh` running xcodegen). Never edit it by hand; `project.yml` is the source of truth.
- After adding/removing/renaming files under `App/`, `ShareExtension/`, or `AppTests/`, regenerate: `xcodegen generate` (the scripts already do this) — then **commit the resulting `project.pbxproj` change** alongside the source change, or CI/fresh checkouts won't see the new files. Leave `xcuserdata` (Xcode UI state) out of commits. Sources are globbed, so you don't list files in `project.yml`; you only edit `project.yml` to change targets/settings/dependencies.

## Persistence & storage (no SwiftData / Core Data)

The catalogue is sourced entirely from per-entry **folders on disk** — the single source of truth:
- Each imported file gets a folder under the active root; `metadata.json` (`EntryMetadata`) is the synced sidecar, and a local-only `resources/` subfolder caches extracted/downloaded icons and photos (plus a `manifest.json` and `remote-hrefs.json`).
- `Catalog` (`@MainActor @Observable`) holds the in-memory list, rebuilt from disk by `CatalogScanner`. The parsed KML tree is **re-parsed from disk on open** via `PinfoldCore`, never persisted.
- App preferences (`AppSettings`) are `UserDefaults`-backed and per-device.
- **Tests:** inject a temporary `StorageLocations(root:)` under `FileManager.temporaryDirectory` so tests never touch real Application Support or iCloud. Suites that share mutable `@MainActor` state (e.g. a `Catalog`) are marked `@Suite(.serialized)`.

## Architecture

MV + `@Observable` (no view-model layer); see **Persistence & storage** for the data model. Heavy import work runs off-main in `ImportService.prepare` (returns a `Sendable` `ImportResult`); `commit` and all storage mutation are `@MainActor`. Original files + the synced `metadata.json` live under the active root (local Application Support, or the iCloud container's `Documents` when sync is on); the derivable `resources/` cache is always local. Files arrive three ways — the in-app `fileImporter`, the "Open in Pinfold" file-type association (`onOpenURL` → `Documents/Inbox`), and the share extension (App Group inbox) — all converging on `ImportService`.

## Releases

- **Versioning:** `MARKETING_VERSION` in `project.yml` is the single source — both Info.plists read `$(MARKETING_VERSION)`. Before cutting a release, verify the version against what is actually **live or in review on App Store Connect**, not the repo (versions have shipped untagged before; reusing a released version fails Xcode Cloud's "Prepare Build for App Store Connect" with ITMS-90186 "train closed" while every log line stays green). Bump → `xcodegen generate` → commit `project.yml` + `project.pbxproj` together (`chore: bump marketing version to X.Y`).
- **Build numbers:** Xcode Cloud manages `CFBundleVersion` per build — never bump `CURRENT_PROJECT_VERSION` for a release.
- **Tag every shipped version** so the repo records what's live: annotated tag `vX.Y` on the release commit on `main`, carrying the release notes, plus a GitHub release titled `Pinfold X.Y` (`gh release create vX.Y --title "Pinfold X.Y" --notes-file …`).
- **Release notes:** user-facing, bold-led paragraphs (see the v1.4/v1.7 releases for the voice). App Store Connect's "What's New" field is plain text — strip the markdown before pasting.
- **Order of operations:** merge to `main` → bump version → tag + GitHub release → let Xcode Cloud build `main` (a failed "Prepare Build for App Store Connect" with green logs is usually the closed-train version collision above, or a transient — the real error arrives by email as ITMS-xxxx).

## Conventions

- Git: branch for non-trivial work and merge to `main` with `--no-ff`; **commit only when explicitly asked**.
- The App Group / share-extension handoff only works on a **signed** build — it's inactive on the unsigned simulator builds (the app guards on `AppGroup.inboxURL != nil`).
- SwiftFormat's `redundantSendable` rule strips explicit `Sendable` from internal all-value-type structs; the conformance is then implicit and compiler-enforced, so don't re-add it to fight the formatter.
- Keep `Task.detached` closures inside nonisolated free functions and keep view `body`s small — large closures captured in big view bodies hit a Swift SIL `ClosureLifetimeFixup` pathology (see the comments in `FavoritesView.swift`).
