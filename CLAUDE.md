# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Pinfold is an offline KML/KMZ catalogue app for iOS/iPadOS/macOS (Designed for iPad), min **iOS 26**, Liquid Glass UI. Two parts:
- `PinfoldCore/` — a standalone Swift Package (no UIKit/SwiftUI/SwiftData): the KML/KMZ parser. Entry point `KMLReader.read(data:) -> ParsedKML`. Depends on ZIPFoundation.
- `App/`, `ShareExtension/`, `AppTests/` — the app, built into an Xcode project **generated by XcodeGen** from `project.yml`.

Design + plans live in `docs/superpowers/`. Sample KML/KMZ files are in `samples/`.

## Build & test — use the scripts, not raw xcodebuild

- App build: `scripts/build.sh` · App tests: `scripts/test.sh` (filter: `scripts/test.sh -only-testing:PinfoldTests/<Suite>`).
  These target the **iPhone 17 simulator on iOS 26.5 with code signing disabled** — the only known-good destination here (the device/team isn't configured, and only the 26.5 simulator runtime is installed).
- Parser package tests: `cd PinfoldCore && swift test`.

## Formatting & linting

- **Format:** `swiftformat` (config-less, pinned via `.swift-version`). A PostToolUse hook auto-formats each `.swift` file Claude edits — let it own comma/spacing style.
- **Lint:** `swiftlint lint` (config in `.swiftlint.yml`). Keep it at **0 errors**; the existing warnings are a known backlog. `trailing_comma` is disabled there on purpose (SwiftFormat owns comma style). Annotate genuinely-intentional violations inline (e.g. `// swiftlint:disable:next force_try -- <reason>`) rather than loosening a rule globally.

## XcodeGen workflow

- `Pinfold.xcodeproj` is **generated and gitignored** — never edit it by hand, and don't commit it. `project.yml` is the source of truth (committed).
- After adding/removing/renaming files under `App/`, `ShareExtension/`, or `AppTests/`, regenerate: `xcodegen generate` (the scripts already do this). Sources are globbed, so you don't list files in `project.yml`; you only edit `project.yml` to change targets/settings/dependencies.

## SwiftData rules (these cause silent EXC_BREAKPOINT crashes if violated)

- **Tests:** every SwiftData suite must share the one `TestModelContainer.shared` (AppTests/TestSupport.swift) via `TestModelContainer.freshContext()`, and be marked `@Suite(.serialized)`. Creating a second `ModelContainer` for the same models in one process traps.
- Do **not** use `isStoredInMemoryOnly: true` (traps on the iOS 26.5 sim — use an on-disk `ModelConfiguration(url:)`), and do **not** put `@Attribute(.unique)` on model identifiers (its upsert path traps).

## Architecture

MV + `@Observable` (no view-model layer). SwiftData persists catalogue metadata only (`KMLFileEntry`, `AppSettings`); the parsed KML tree is **re-parsed from disk on open** via `PinfoldCore`, never stored. Heavy import work runs off-main in `ImportService.prepare` (returns a `Sendable` value); `commit` and all SwiftData access are `@MainActor`. Original files + cached resources live under Application Support, one folder per import.

## Conventions

- Git: branch for non-trivial work and merge to `main` with `--no-ff`; **commit only when explicitly asked**.
- The App Group / share-extension handoff only works on a **signed** build — it's inactive on the unsigned simulator builds (the app guards on `AppGroup.inboxURL != nil`).
